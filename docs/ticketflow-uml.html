3. Diagrammes de séquence

3.1 — Créer un ticket (US-04)

Le client authentifié crée un ticket dans son organisation. Le système génère un numéro séquentiel, enregistre l'historique et notifie les agents via Celery.

Séquence — Création d'un ticket par un client

3.2 — Changer le statut d'un ticket (US-06)

L'agent demande une transition de statut. Le service délègue la validation à la machine à états avant de persister.

Séquence — Transition de statut par un agent

3.3 — S'inscrire (US-01)

Un visiteur crée un compte avec email et mot de passe. L'API retourne une paire de tokens JWT (access + refresh).

Séquence — Inscription d'un utilisateur

3.4 — Créer une organisation (US-02)

Un utilisateur authentifié crée une organisation. Il en devient automatiquement ADMIN via un Membership créé dans la même transaction.

Séquence — Création d'une organisation

3.5 — Inviter un membre (US-03)

L'admin invite un utilisateur par email. Si l'utilisateur existe déjà, un Membership est créé directement. Sinon, une invitation par email est envoyée via Celery.

Séquence — Invitation d'un membre par un admin

3.6 — Lister et filtrer les tickets (US-05)

L'agent consulte les tickets de son organisation avec filtres (statut, priorité, assignation), recherche textuelle et pagination.

Séquence — Liste paginée et filtrée des tickets

3.7 — Assigner un ticket (US-07)

Un agent ou admin assigne un ticket à un agent de l'organisation. Le service vérifie que l'agent cible est bien membre de l'orga avec le rôle AGENT ou ADMIN.

Séquence — Assignation d'un ticket à un agent

3.8 — Ajouter un commentaire (US-08)

Un utilisateur ajoute un commentaire à un ticket. Le flag is_internal est réservé aux agents/admins. Les clients ne voient pas les notes internes.

Séquence — Ajout d'un commentaire avec flag is_internal

3.9 — Consulter l'historique d'un ticket (US-09)

Un utilisateur consulte la timeline d'un ticket : changements de statut, assignations, commentaires. Les notes internes sont filtrées pour les clients.

Séquence — Consultation de l'historique / timeline

4. Diagramme d'états — Cycle de vie d'un ticket

Diagramme d'états-transitions — Statut du ticket
🧠
Implémentation en Django La machine à états sera implémentée comme un dictionnaire TRANSITIONS dans le service layer, pas comme un champ model. Chaque clé est l'état source, la valeur est un set d'états cibles. La méthode TicketService.transition() vérifie la validité avant de persister. Les transitions en rouge (réouverture) ont des restrictions de rôle supplémentaires. Détail des transitions autorisées dans le Cahier des Charges §6.
