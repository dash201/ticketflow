TicketFlow

Système de gestion de tickets de support client multi-tenant — Cahier des charges complet. L'analyse UML et la modélisation de données (MCD/MLD) sont disponibles dans des documents séparés.

Projet portfolio — API DRF • Multi-tenant • JWT • Docker

1. Contexte & Vision

1.1 — Le problème

Les PME et startups ont besoin d'un système de support client pour centraliser les demandes de leurs utilisateurs, mais les solutions existantes (Zendesk, Freshdesk, Intercom) sont soit trop coûteuses, soit trop complexes pour leurs besoins. Il n'existe pas de solution open-source légère, multi-tenant, et exposant une API REST propre pour être intégrée à n'importe quel frontend ou outil tiers.

1.2 — La solution

TicketFlow est une API REST de gestion de tickets de support client. Chaque organisation dispose de son espace isolé (multi-tenant) avec ses agents, ses clients, ses tickets et ses paramètres. L'API est conçue pour être consommée par n'importe quel frontend (React, Vue, mobile) ou intégrée dans un workflow tiers via webhooks.

🎯
Positionnement portfolio Ce projet démontre la maîtrise de : l'architecture multi-tenant, la conception d'API REST avec DRF, l'authentification JWT, le workflow métier (machine à états), les tests automatisés, le déploiement Docker, et la documentation Swagger/OpenAPI.

2. Objectifs

2.1 — Objectifs fonctionnels

OF-1 : Gestion multi-tenant

Chaque organisation vit dans un espace isolé. Un utilisateur ne voit jamais les données d'une autre organisation. L'isolation est garantie au niveau ORM (QuerySet filtering).

MUST

OF-2 : CRUD des tickets

Créer, lire, mettre à jour, filtrer, rechercher et supprimer des tickets. Chaque ticket a un sujet, une description, une priorité, un statut et un assigné.

MUST

OF-3 : Workflow de statuts

Les tickets suivent un workflow contrôlé : New → Open → Pending → Resolved → Closed. Les transitions invalides sont rejetées par l'API.

MUST

OF-4 : Commentaires & historique

Agents et clients peuvent commenter les tickets. Chaque changement de statut ou d'assignation est tracé dans un historique horodaté.

MUST

OF-5 : Assignation des agents

Un ticket peut être assigné à un agent de l'organisation. L'assignation peut être manuelle ou par défaut (round-robin optionnel).

MUST

OF-6 : Auth & rôles

Authentification JWT (access + refresh + rotation). 3 rôles par organisation : Admin, Agent, Client. Permissions granulaires par endpoint.

MUST

2.2 — Objectifs techniques

OT-1 : API REST conforme

Endpoints RESTful, pagination, filtrage, tri, recherche. Réponses JSON standardisées. Documentation OpenAPI/Swagger auto-générée.

Couverture >80%. Tests unitaires (models, services), tests d'intégration (endpoints, permissions), tests d'isolation multi-tenant.

3. Acteurs du système

🔑 Super Admin

Administrateur de la plateforme. Gère les organisations, peut accéder à toutes les données. Correspond au is_superuser Django.

Hors scope API publique — accès admin Django uniquement.

👔 Admin (Organisation)

Propriétaire/administrateur d'une organisation. Gère les agents, configure les paramètres de l'orga, voit toutes les statistiques et tous les tickets de son espace.

Rôle : ADMIN

🛠 Agent

Membre de l'équipe support. Traite les tickets, change les statuts, commente, peut être assigné. Voit tous les tickets de son organisation.

Rôle : AGENT

👤 Client

Utilisateur final qui soumet des demandes de support. Ne voit que ses propres tickets. Peut commenter pour apporter des précisions.

Rôle : CLIENT

🤖 Système

Acteur non-humain. Déclenche les notifications, les changements automatiques (timeout, SLA), les tâches Celery.

Pas de rôle — exécution interne

🧠
Isolation multi-tenant Un Admin, Agent ou Client est toujours rattaché à une organisation via la table Membership. L'organisation agit comme filtre implicite sur toutes les requêtes ORM — un agent de l'orga A ne voit jamais les tickets de l'orga B.

5. User Stories

5.1 — Inscription & Organisation

US-01 MUST

En tant qu'utilisateur non inscrit, je veux créer un compte avec mon email et un mot de passe, afin d'accéder à la plateforme.

  • Le mot de passe doit respecter les règles Django (min 8 chars, pas trop commun, pas entièrement numérique)
  • L'email doit être unique
  • La réponse contient les tokens JWT (access + refresh)
US-02 MUST

En tant qu'utilisateur authentifié, je veux créer une organisation, afin de commencer à recevoir des tickets de support.

  • L'organisation a un nom et un slug unique (auto-généré ou personnalisé)
  • Le créateur obtient automatiquement le rôle ADMIN dans cette organisation
  • Un utilisateur peut appartenir à plusieurs organisations
US-03 MUST

En tant qu'admin d'une organisation, je veux inviter des agents et des clients par email, afin de constituer mon équipe de support et ma base de clients.

  • L'invitation spécifie le rôle (AGENT ou CLIENT)
  • Si l'email correspond à un compte existant, le membership est créé directement
  • Si l'email est inconnu, un lien d'invitation est envoyé

5.2 — Gestion des tickets

US-04 MUST

En tant que client, je veux créer un ticket de support avec un sujet, une description et une priorité, afin de signaler un problème ou poser une question.

  • Le ticket reçoit un numéro séquentiel (TF-001)
  • Le statut initial est NEW
  • L'organisation est déduite du membership actif du client
  • Les agents de l'orga sont notifiés (si notifications activées)
US-05 MUST

En tant qu'agent, je veux voir la liste de tous les tickets de mon organisation avec des filtres (statut, priorité, assigné), afin de prioriser mon travail.

  • Pagination par 20 (configurable)
  • Filtrage par : statut, priorité, assigné, date de création
  • Tri par : date, priorité, dernière mise à jour
  • Recherche textuelle sur sujet et description
US-06 MUST

En tant qu'agent, je veux assigner un ticket à moi-même ou à un collègue, afin de clarifier la responsabilité du traitement.

  • L'assigné doit être un agent ou admin de la même organisation
  • Le changement d'assignation est tracé dans l'historique
  • L'agent assigné est notifié
US-07 MUST

En tant qu'agent, je veux changer le statut d'un ticket selon le workflow défini, afin de refléter l'avancement du traitement.

  • Seules les transitions valides sont autorisées (cf. diagramme d'états dans l'analyse UML)
  • La transition est tracée dans l'historique avec ancien et nouveau statut
  • Le client est notifié du changement

5.3 — Commentaires

US-08 MUST

En tant que client ou agent, je veux ajouter un commentaire à un ticket, afin d'apporter des précisions ou une réponse.

  • Le commentaire est horodaté et identifie l'auteur
  • Un agent peut marquer un commentaire comme "interne" (invisible pour le client)
  • Les participants sont notifiés
US-09 MUST

En tant que client, je veux voir l'historique complet de mon ticket (commentaires + changements de statut), afin de suivre l'avancement de ma demande.

  • Les commentaires internes sont exclus pour le client
  • L'historique est trié chronologiquement
  • Chaque entrée affiche le type (commentaire, changement de statut, assignation)

7. Exigences non-fonctionnelles

🔒 Sécurité

Isolation multi-tenant vérifiée par tests automatisés (un user ne peut jamais accéder aux données d'une autre orga). Protection OWASP : pas de SQL injection (ORM), pas de mass assignment (serializer fields explicites), pas d'IDOR (QuerySet filtré par orga), rate limiting (throttling DRF). JWT avec rotation du refresh token et blacklist.

⚡ Performance

Temps de réponse API < 200ms pour les endpoints de liste (avec pagination). Utilisation de select_related / prefetch_related pour éviter les N+1. Index DB sur les colonnes filtrées (statut, priorité, organisation, assigné, created_at).

🧪 Testabilité

Couverture > 80%. Tests d'isolation multi-tenant dédiés. Factory Boy pour les fixtures. Tests paramétrés pour les permissions (chaque combinaison rôle × action). CI qui bloque le merge si les tests échouent.

📖 Maintenabilité

Service layer pour la logique métier (pas de fat views/models). Code formaté (ruff). Type hints sur les signatures publiques. Documentation docstring sur les services.

📦 Déployabilité

Un docker-compose up lance tout l'environnement (web + db + redis). Variables d'environnement pour la configuration (pas de secrets dans le code). GitHub Actions pour CI/CD.

📄 Documentation

Swagger/OpenAPI auto-généré par drf-spectacular. README avec instructions d'installation, d'utilisation et de contribution. Exemples de requêtes cURL pour chaque endpoint.

18.2 — Architecture en couches

Client HTTP → DRF View → Serializer → Service Layer → ORM (Models) → PostgreSQL
✅
Principe directeur Les Views sont thin : elles délèguent au Service Layer. Les Models sont thin : ils déclarent le schéma et les managers custom. Toute la logique métier (validation de transition, génération de numéro, notification) vit dans le Service Layer. C'est le pattern enseigné dans le M6 du cours Django.

18.3 — Multi-tenant : stratégie d'isolation

🏢
Shared Database, Shared Schema Toutes les organisations partagent la même base de données et les mêmes tables. L'isolation est garantie par le filtrage ORM : chaque QuerySet est filtré par organization_id. C'est la stratégie la plus simple à implémenter et suffisante pour une application de cette échelle. L'alternative (schéma par tenant ou DB par tenant) est overkill ici.

Chaque vue qui manipule des données tenant-scoped récupère l'organisation depuis l'URL ({slug}) et la passe au service layer, qui l'injecte dans tous les QuerySets :

# Dans chaque service
def list_tickets(*, organization: Organization, ...) -> QuerySet:
    return Ticket.objects.filter(
        organization=organization,
        is_deleted=False
    )
