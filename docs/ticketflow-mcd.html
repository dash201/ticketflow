MCD — Entités et associations

2. Règles de gestion

RG-1 : Membership

Un USER peut appartenir à plusieurs ORGANISATIONs via un MEMBERSHIP. Un MEMBERSHIP porte un rôle (ADMIN, AGENT, CLIENT). La combinaison (user, organization) est unique.

RG-2 : Ticket

Un TICKET est toujours rattaché à exactement une ORGANISATION et un USER créateur. L'assignation à un agent est optionnelle (0,1).

RG-3 : Comment

Un COMMENT appartient à exactement un TICKET. Le booléen is_internal détermine la visibilité (agents seulement vs. tous).

RG-4 : Historique

Chaque changement de champ important (status, priority, assigned_to) sur un TICKET génère une entrée HISTORIQUE avec les valeurs avant/après.

RG-5 : Numérotation

Le numéro de TICKET est séquentiel par ORGANISATION (pas global). Deux orgas peuvent avoir chacune un ticket TF-001.

RG-6 : Soft delete

La suppression est toujours logique (soft delete via is_deleted). Aucune donnée n'est physiquement supprimée de la base.

3. Modèle Logique de Données (MLD)

Passage du MCD au MLD : chaque entité devient une table, chaque association N:M devient une table intermédiaire, les clés étrangères matérialisent les associations 1:N.

users_user

id UUID PK | email VARCHAR(254) UNIQUE NOT NULL | password VARCHAR(128) NOT NULL | first_name VARCHAR(150) | last_name VARCHAR(150) | is_active BOOLEAN DEFAULT TRUE | is_staff BOOLEAN DEFAULT FALSE | date_joined TIMESTAMPTZ NOT NULL

organizations_organization

id UUID PK | name VARCHAR(100) NOT NULL | slug VARCHAR(100) UNIQUE NOT NULL | owner_id UUID FK → users_user(id) NOT NULL | auto_assign BOOLEAN DEFAULT FALSE | created_at TIMESTAMPTZ NOT NULL | updated_at TIMESTAMPTZ NOT NULL

organizations_membership

id UUID PK | user_id UUID FK → users_user(id) NOT NULL | organization_id UUID FK → organizations_organization(id) NOT NULL | role VARCHAR(10) CHECK(IN 'ADMIN','AGENT','CLIENT') NOT NULL | joined_at TIMESTAMPTZ NOT NULL

UNIQUE(user_id, organization_id) | INDEX(organization_id, role)

tickets_ticket

id UUID PK | organization_id UUID FK → organizations_organization(id) NOT NULL | number INTEGER NOT NULL | subject VARCHAR(255) NOT NULL | description TEXT NOT NULL | status VARCHAR(10) CHECK(IN 'NEW','OPEN','PENDING','RESOLVED','CLOSED') DEFAULT 'NEW' | priority VARCHAR(10) CHECK(IN 'LOW','MEDIUM','HIGH','URGENT') DEFAULT 'MEDIUM' | created_by_id UUID FK → users_user(id) NOT NULL | assigned_to_id UUID FK → users_user(id) NULL | created_at TIMESTAMPTZ NOT NULL | updated_at TIMESTAMPTZ NOT NULL | is_deleted BOOLEAN DEFAULT FALSE

UNIQUE(organization_id, number) | INDEX(organization_id, status) | INDEX(organization_id, priority) | INDEX(assigned_to_id) | INDEX(created_at DESC)

tickets_comment

id UUID PK | ticket_id UUID FK → tickets_ticket(id) NOT NULL | author_id UUID FK → users_user(id) NOT NULL | body TEXT NOT NULL | is_internal BOOLEAN DEFAULT FALSE | created_at TIMESTAMPTZ NOT NULL | updated_at TIMESTAMPTZ NOT NULL | is_deleted BOOLEAN DEFAULT FALSE

INDEX(ticket_id, created_at) | INDEX(author_id)

tickets_tickethistory

id UUID PK | ticket_id UUID FK → tickets_ticket(id) NOT NULL | changed_by_id UUID FK → users_user(id) NOT NULL | field VARCHAR(50) NOT NULL | old_value VARCHAR(255) NULL | new_value VARCHAR(255) NOT NULL | created_at TIMESTAMPTZ NOT NULL

INDEX(ticket_id, created_at)

⚠️
Isolation multi-tenant L'isolation n'est pas une contrainte DB mais une contrainte applicative : chaque QuerySet est filtré par organization_id dans le service layer. Des tests d'intégration dédiés vérifient qu'un utilisateur de l'orga A ne peut jamais accéder aux données de l'orga B. Voir la stratégie détaillée dans le Cahier des Charges §18.
🧠
Passage du MLD au code Django Chaque table du MLD correspond à un Model Django. Les UUID PK se déclarent avec models.UUIDField(primary_key=True, default=uuid.uuid4). Les ENUM se déclarent avec models.TextChoices. Les UNIQUE TOGETHER se déclarent dans class Meta: constraints. Les INDEX se déclarent dans class Meta: indexes. Le diagramme de classes complet est dans l'Analyse UML §2.
